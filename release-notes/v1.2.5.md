## Highlights

This release turns the Reference field into full relations between your content, the way Strapi does it. It adds a new Reference Page type whose web address you define yourself, and it gives reference data like tags and cities its own place in Content.

## What's new

- Relations between content: a Reference field now links one entry to others. You can pick a single linked entry (like an author) or many (like tags), manage the link from either side, and point a field at one content type or several at once. This covers every common relationship, one-to-one, one-to-many, many-to-many, and one-directional links.
- A relation picker in the editor: Reference fields now show a searchable picker of existing entries instead of a plain text box. Selected entries appear as removable chips, in order, with a type label when a field can point at more than one type.
- Reference Page type with custom URLs: a new page type for reference data such as tags or cities. You set its public web address with a pattern, for example `/blogs/tags/{slug}` gives `/blogs/tags/common-problems`, and `/appliance-repair/{slug}` gives `/appliance-repair/bridgeport-ct`. Use `{slug}` where the entry slug goes, plus `{locale}` if you need it.
- A Reference tab in Content: reference data gets its own tab, with a separate sub-tab for each reference type (Tags, Cities, and so on). The tab appears once you create your first Reference Page type.
- Linked content comes back expanded: when your site reads an entry through the API, its relations are returned as the full linked entries (id, slug, fields), so you can render a post's tags or author directly without extra requests.

## Fixes and improvements

- Reference data stays out of the way: tags, cities and other reference entries no longer appear in the All Content list, its type filter, or its counts. They live under the Reference tab instead.
- General stability and performance improvements.
