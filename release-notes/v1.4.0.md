## v1.4.0 — least-privilege Docker access for the updater

This completes the self-host hardening line (v1.3.6–v1.3.9): the updater can now run with **no
direct Docker socket**, via a scoped, read-only socket proxy, so a compromised updater can no
longer perform arbitrary host-root operations.

The last blocker was backups. The updater used to shell out to Docker to run `pg_dump`/`psql`
(`exec`, then a streaming `run`); both need a connection the proxy can't forward, so backups, and
therefore upgrades, would hang under the proxy. Fixed:

- **Backups/restores no longer use Docker at all.** The updater now bundles the Postgres client
  and connects to the database **directly over the network**. Verified end to end: the dump is
  complete and restorable, and it runs cleanly with the updater pointed at the socket proxy.
- **External/managed databases can now be backed up too** (it connects over `DATABASE_URL`), which
  the old "exec into the local container" approach could never do.
- The updater shares the internal `backend` network with Postgres for these network backups.

The scoped `socket-proxy` is now **fully functional** (opt-in via `--profile secure-socket`). Once
it's validated on a live upgrade it will become the default, so every install gets least-privilege
Docker access out of the box.
