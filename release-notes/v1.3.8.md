## Production hardening of the self-host stack

A full infrastructure-review pass over `docker-compose.prod.yml`. None of this changes how the
app behaves; it makes the stack safe to run 24/7 unattended. The headline fix closes the second
half of the disk-full class of incident: **container logs are now capped, so they can't silently
fill the disk** (the updater's image pruning, shipped in 1.3.6–1.3.7, was the other half).

- **Log rotation on every service** (10MB × 5 files max). Unbounded Docker logs were a latent
  disk-fill source independent of image buildup.
- **Memory + CPU limits on every service**, all env-tunable, so one runaway container (leak, bad
  query, big import) can't OOM the whole host. Sensible defaults fit a 2–4 GB VPS; raise them on
  bigger hosts via `.env` (`API_MEM_LIMIT`, `PG_MEM_LIMIT`, …).
- **Capability hardening**: `cap_drop: ALL` with only the minimum added back per service,
  `no-new-privileges`, and `init: true` (clean PID 1 / zombie reaping) everywhere.
- **Healthchecks on every service** (previously caddy, the updater, redis and meilisearch had
  none), with graceful shutdown (`stop_grace_period`, Postgres `SIGINT` fast-shutdown) so updates
  and restarts don't risk half-written state.
- **Two-tier network**: Postgres/Redis/Meilisearch now sit on an `internal` network with no route
  to the internet; only the edge tier is reachable.
- **Postgres**: `shm_size: 256mb` (prevents "could not resize shared memory" under parallel
  queries) and higher file-descriptor limits.
- **Pinned image versions** (caddy 2.8, postgres 16.6, redis 7.4) instead of moving tags.
- **Opt-in least-privilege Docker access**: a new `--profile secure-socket` runs a scoped,
  read-only `docker-socket-proxy` in front of the updater, so the updater no longer needs the raw
  host socket. Off by default (the direct socket remains the default, unchanged); see the updater
  notes in the compose file and `.env` example to enable it.

Every change was validated with `docker compose config`, and the capability-dropped Postgres,
Caddy and Redis images were smoke-tested to confirm they still start and stay healthy.

> Existing servers pick all of this up on the next `curl -fsSL https://get.flowcms.co/install.sh |
> sudo bash` (idempotent; recreates the stack with the new compose). New installs get it from day
> one. Defaults are unchanged for the updater's socket and behavior, so the upgrade is safe.
